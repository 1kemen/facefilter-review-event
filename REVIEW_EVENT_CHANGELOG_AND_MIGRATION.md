# 리뷰 이벤트 MVP 변경 내역 및 디자인 이식 검토

이 문서는 페이스필터 천호점 리뷰 이벤트 MVP의 현재 상태, 사용자가 요청한 주요 변경사항, 실제 반영된 내용, 그리고 외부 디자인 시안을 실제 코드에 이식할 때의 주의사항을 정리한 문서입니다.

## 1. 결론

디자인 이식은 가능합니다.

다만 **완성된 `index.html` / `styles.css`를 그대로 덮어쓰는 방식은 권장하지 않습니다.**

현재 앱은 단순 정적 페이지가 아니라 `index.html`의 특정 `id`, `name`, `data-*` 속성을 `app.js`가 직접 잡아서 동작하는 구조입니다.  
따라서 외부 시안의 HTML/CSS를 그대로 복붙하면 아래 기능이 깨질 수 있습니다.

- 고객 참여 등록
- 리뷰 작성 완료 처리
- 상품뽑기
- 카카오톡 채널 혜택
- 직원/관리자 로그인
- 직원 지급완료 처리
- Supabase 연동
- 세션 복구
- 중복 참여 차단

안전한 방식은 **외부 디자인을 참고해서 현재 `index.html` 구조에 맞게 재구성**하는 것입니다.

7. 물방울 선택
8. 선택한 물방울로 상품뽑기 확정
9. 카카오톡 채널 추가 혜택 선택
10. 직원 확인 화면

## 5. 사용자가 요청한 주요 내용

### 리뷰 이벤트 자동화

- 기존 수기 플로우를 QR 기반으로 전환
- 고객이 직접 포토리뷰 작성 후 뽑기 진행
- 직원은 최종 화면과 실제 리뷰 화면을 확인한 뒤 지급

### 참여 제한 및 어뷰징 방지

- 네이버 ID/닉네임 기준 4주 간격 참여 제한
- 같은 이름 + 전화번호 + 리뷰 닉네임/ID일 때만 기존 세션 복구
- 지급완료된 참여자는 고객 화면에서 결과 재오픈 차단
- 새로고침, 뒤로가기, 캐시 초기화로 재뽑기 불가
- 같은 QR 링크를 여러 명이 접속해도 각자 세션 분리

### 리뷰 작성 복귀 문제 대응

- 네이버 리뷰 페이지 이동 후 고객이 이벤트 화면으로 돌아오기 어려움
- QR을 다시 찍고 같은 정보를 입력하면 기존 진행 상태로 복구
- 리뷰 작성 완료 버튼을 별도 노출
- 10초 자동 뽑기 가능 로직 제거

### 상품뽑기 안전장치

- 기존에는 물방울 터치 한 번으로 결과가 확정될 수 있어 위험했음
- 현재는 물방울 선택 후 큰 버튼을 한 번 더 눌러야 확정
- 서버에서도 리뷰 작성 완료 + 물방울 번호 없이는 뽑기 거절하도록 SQL 패치 제공

### 카카오톡 채널 혜택

- 카카오톡 채널 추가는 선택 혜택
- 혜택명은 애프터케어 시트팩 1매
- 고객이 배너/버튼을 누르면 혜택 신청으로 기록
- 이미 채널 추가한 고객도 혜택 받을 수 있도록 버튼 제공

### 직원/관리자 화면

- 관리자 페이지에서 참여자 관리
- 직원 페이지에서 현장 지급 관리
- 직원이 담당자명, 고객 차트번호, 지급 메모 입력
- 담당자명은 저장 후 수정 불가
- 미완료 고객은 직원이 리뷰 화면을 직접 확인한 뒤 대신뽑기 가능
- 대신뽑기 시 특이사항 메모 유도
- 관리자 삭제 기능 추가
- 상품 설정 기능 추가
- 감사로그 확인 기능 추가

### 직원 계정

- 팀별 직원 계정 사용 가능
- 예시:
  - 코디팀
  - 어시팀
  - 간호팀
  - 피부팀
  - 상담팀

## 6. 실제 반영된 주요 변경사항

### 고객 입력 화면

- `네이버 ID 또는 리뷰 닉네임`을 `네이버 리뷰 닉네임`으로 변경
- 보조 문구 추가:

```text
리뷰 화면에 보이는 닉네임을 입력해 주세요. 닉네임을 모르면 네이버 ID를 입력해 주세요.
```

- 입력칸을 세로형으로 정리
- 휴대폰/닉네임 2열 배치 제거

### 리뷰 단계

- 리뷰 작성하기 버튼을 누르면 네이버 리뷰 링크로 이동
- 이벤트 화면 복귀 후 `리뷰 작성 완료` 버튼을 눌러야 다음 단계 가능
- 리뷰 링크 클릭 후 10초가 지나면 자동으로 뽑기 가능하던 로직 제거

### 뽑기 단계

- 물방울은 5개 고정
- 물방울 클릭은 결과 확정이 아니라 선택만 수행
- 선택 후 `n번 물방울로 뽑기` 버튼을 눌러야 결과 확정
- 서버 패치:

```text
supabase/patch-require-explicit-review-done-before-draw.sql
```

### 지급완료/세션 복구

- 지급완료된 참여자는 고객 화면에서 같은 정보로 다시 입력해도 기존 결과 payload를 내려주지 않도록 보완
- 관련 패치:

```text
supabase/patch-no-completed-resume-payload.sql
```

### 진단 SQL

문제 발생 시 원인 확인용 SQL 추가:

```text
supabase/forensic-draw-proof.sql
supabase/diagnose-participant-flow-anomaly.sql
```

## 7. 외부 디자인 시안 이식 가능 여부

### 가능한 부분

아래는 비교적 안전하게 이식 가능합니다.

- 색상 토큰
- 폰트 크기
- 카드/패널 스타일
- 버튼 스타일
- 여백
- 배경 질감
- 모달 디자인
- 진행바 디자인
- 고객 화면의 전체 분위기
- 직원/관리자 화면의 표/패널 스타일

이 부분은 대부분 `styles.css` 중심으로 적용 가능합니다.

### 주의가 필요한 부분

아래는 `index.html` 구조 변경이 필요할 수 있습니다.

- 인트로 화면 추가
- 슬롯식 입력 UI
- 단계별 카드 구조 변경
- 리뷰 작성 안내 모달 구조 변경
- 물방울 캐릭터/이미지 추가
- 결과 공개 카드 레이아웃 변경

이때도 기존 `id`, `name`, `data-*` 속성은 유지해야 합니다.

### app.js 수정이 필요한 부분

아래는 CSS만으로는 불가능하고 `app.js` 수정이 필요합니다.

- 입력칸 자동 전환
- 슬라이드 전환 애니메이션을 상태와 연결
- 결과 공개 애니메이션을 실제 뽑기 완료 시점과 연결
- 에러 화면을 상태별로 분리
- 복구 화면을 별도 UX로 구성
- 리뷰 작성 상태별 버튼 교체를 더 복잡하게 변경

## 9. 권장 이식 전략

### 1단계: CSS 토큰화

외부 시안의 컬러, 폰트, 그림자, radius, spacing을 `styles.css` 상단 변수로 정리합니다.

### 2단계: 고객 화면만 먼저 적용

먼저 고객 QR 화면만 디자인을 맞춥니다.

우선순위:

1. 참여 정보 화면
2. 리뷰 작성 화면
3. 상품뽑기 화면
4. 직원 확인 화면
5. 카카오톡 혜택 모달

### 3단계: 관리자/직원 화면 적용

고객 화면 안정화 후 직원/관리자 화면을 정리합니다.

### 4단계: app.js 변경은 최소 단위로

슬롯 입력, 슬라이드 전환, 결과 애니메이션처럼 기능 상태와 연결되는 작업은 별도 작업으로 분리합니다.

## 10. 다른 에이전트에게 전달할 핵심 문장

```text
완성된 시안의 index.html/styles.css를 그대로 덮어쓰지 말고, 현재 index.html의 id/name/data-* 구조를 유지한 채 styles.css 중심으로 이식하세요.
app.js는 기능 로직이므로 디자인만 변경할 때는 건드리지 마세요.
구조 변경이 필요하면 기존 id와 form name을 반드시 유지하고, 이벤트 연결이 깨지는지 확인하세요.
```

## 11. 현재 판단

외부 디자인 정리 문서는 방향성 참고용으로는 유용합니다.

하지만 그대로 복붙하는 이식은 위험합니다.

실제 적용은 아래 방식이 안전합니다.

```text
외부 시안 = 디자인 기준서
현재 index.html = 기능 구조 기준서
현재 styles.css = 실제 이식 대상
app.js = 변경 최소화
```
